# AionUi UI Context

This file is the canonical AionUi UI context for both AI agents and contributors.

## How to use this file

- Read this file before implementing non-trivial UI changes
- Treat this as the UI-layer supplement to `.claude/skills/architecture/SKILL.md`
- Use `architecture` skill for file placement, naming, shared-vs-private boundaries, Arco/CSS technical rules
- Use this file for AionUi-specific UI decisions: reuse strategy, AOU wrappers, theme behavior, workflow, and PR expectations

## Design goals

- Product tone: practical, precise, restrained
- Target experience: high information density with low cognitive noise
- Priority: consistency, maintainability, reusability

## Style preferences

- AionUi generally prefers slightly softer and more rounded corners than raw Arco defaults
- When extending or wrapping components, prefer mildly rounded surfaces if they remain consistent with nearby patterns
- Keep corner radius usage consistent within the same feature; avoid mixing sharp and highly rounded surfaces without intent

## Hard constraints

- Do not hard-code color, radius, shadow, or spacing when an existing token/pattern exists
- UI must support both light and dark themes and align with Arco theme behavior
- Interactive states must be complete: `default`, `hover`, `focus`, `active`, `disabled`, `loading`, `error`, `empty`
- User-facing copy must use i18n keys and tolerate longer translations
- Do not stop at desktop-only implementation; consider mobile behavior

## Component library strategy

- Default: Aion base + Arco
- Optional: `shadcn/ui` only when existing Aion/Arco patterns cannot reasonably satisfy the feature
- If `shadcn/ui` is used, adapt it to AOU tokens, spacing rhythm, theme behavior, and interaction patterns
- Do not use `shadcn/ui` as a separate visual system layered on top of AionUi

## Reuse priority

1. Reuse an existing component directly
2. Reuse the closest component with light extension
3. Compose with Aion base + Arco + tokens
4. Create a new visual pattern only if 1-3 cannot satisfy the requirement

## Component selection strategy

- Prefer AOU wrapper components from `src/renderer/components/base/*`
- Then use Arco primitives
- If still insufficient, create a thin reusable AOU wrapper instead of scattering one-off implementations
- For settings nested dialog patterns, also check `src/renderer/components/SettingsModal/index.tsx`

Wrap Arco into an AOU component when any 2 are true:

- Same style pattern repeats across 3+ pages
- Same token/class combination repeats often
- Light/dark behavior needs to stay consistent
- Interaction and accessibility behavior should be standardized

## AI UI Workflow

1. `normalize`
2. `adapt`
3. `harden`
4. `polish`

## Suggested spacing rhythm

AionUi currently does not have dedicated spacing tokens. Prefer the existing rhythm:

- Micro: `4`, `6`
- Default: `8`
- Medium: `10`, `12`
- Section: `16`
- Container: `20`, `24`, `28`, `32`

## PR expectations for UI changes

UI-related PRs should include:

- `Reference Components`: at least 2 existing AionUi references
- Reuse rationale: what was reused, what was extended, and why
- Desktop + mobile screenshots
- A short self-check summary for `normalize`, `adapt`, `harden`, and `polish`

## UI review rejection triggers

Request changes when any of the following is true:

- No reference components or reuse rationale
- Hard-coded visual values where tokens or patterns already exist
- Missing mobile adaptation evidence
- Missing light/dark support
- Missing empty/error/loading treatment
- i18n or long-text behavior is clearly broken
- A new visual language is introduced without strong justification
